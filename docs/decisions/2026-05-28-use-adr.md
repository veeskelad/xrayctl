# ADR: Use Architecture Decision Records

**Status:** Accepted
**Date:** 2026-05-28
**Tags:** meta

## Context

xrayctl is a new project. Several consequential decisions were made during
scaffold (license, dependency policy, packaging model) and several more will
follow as the v0.1 functionality lands. Without an explicit decision log
those choices live in chat history and commit messages, decay quickly, and
force every future contributor (human or AI) to re-derive context.

## Decision

Adopt **Architecture Decision Records (ADRs)** in
[MADR 3.0](https://adr.github.io/madr/) style, stored in `docs/decisions/`.

- One file per decision, named `YYYY-MM-DD-kebab-slug.md`.
- Mandatory sections: Title, Status, Date, Context, Decision, Consequences.
- Optional: Alternatives considered, References.
- `docs/decisions/INDEX.md` lists every ADR (latest first).
- `docs/decisions/_template.md` is the canonical skeleton for new ADRs.
- New ADR per consequential change: technology / framework / dependency /
  license / packaging / security tradeoff / deprecation.

This first ADR exists to declare the convention itself
([adr-tools](https://github.com/npryce/adr-tools) pattern).

## Consequences

**Positive**
- Every consequential decision has a permanent, searchable home.
- AI assistants can read `INDEX.md` at session start and pick up project
  context without re-deriving it from code.
- New contributors scan one directory to understand the why behind current
  architecture.
- Reverting or revisiting a decision is a Status change with audit trail,
  not a silent rewrite.

**Neutral**
- Adds a documentation step to consequential PRs (see CONTRIBUTING.md).
- Reviewers must check that new dependencies / framework choices are matched
  by an ADR (or argue why they aren't consequential).

**Negative**
- Light upfront tax on a new ADR (~10–20 minutes per decision).
- Risk of ADR-pollution if the bar for "consequential" is too low; mitigated
  by the explicit non-ADR list in CONTRIBUTING.md.
